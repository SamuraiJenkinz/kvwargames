  <action>
Create `src/lib/jsonValidation.ts` exporting:

1. `offsetToLineCol(text: string, offset: number): { line: number; col: number }` — converts a 0-based character offset into 1-based line and column. Walk the string up to `offset`, incrementing `line` on `'\n'` and resetting `col`. Edge cases: offset 0 → `{line: 1, col: 1}`; offset past end → clamp to last position.

2. `parseConfigJson(text: string): { ok: true; value: GameConfig } | { ok: false; error: ParseError }` where `ParseError = { message: string; line: number; col: number; offset: number | null }`.
   - Try `JSON.parse(text)`.
   - On success: do a MINIMAL structural check — verify the parsed value is an object with a `scenarios` array of length ≥ 1 and a `teams` array of length ≥ 1. If structural check fails, return `{ok: false, error: {message: 'Config missing required fields (scenarios, teams)', line: 1, col: 1, offset: null}}`. Do NOT do exhaustive validation — Phase 4 is a facilitator tool, not a hostile-input firewall. Cast the result to `GameConfig` after the structural check.
   - On `JSON.parse` throw: extract the offset from the error message via regex `/at position (\d+)/` (V8 / modern Chromium / Edge / Firefox emit this). If the regex matches, call `offsetToLineCol`. If it doesn't match, return `{ok: false, error: {message: errorMessage, line: 1, col: 1, offset: null}}`.

3. Export the `ParseError` type.

Do NOT install `jsonc-parser`. Native `JSON.parse` + regex is sufficient for the dev-tool target audience. Plan 04-04 may revisit this if cross-browser issues arise.

Create `src/lib/jsonValidation.test.ts` with vitest tests:
- `offsetToLineCol`:
  - offset 0 → `{line: 1, col: 1}`
  - single-line string, offset 5 → `{line: 1, col: 6}`
  - multi-line "ab\ncd\nef", offset 0 → line 1 col 1; offset 3 → line 2 col 1; offset 5 → line 2 col 3; offset 6 → line 3 col 1
  - offset beyond length → clamps to last char
- `parseConfigJson`:
  - `parseConfigJson(JSON.stringify(EDIP_CONFIG))` → `ok: true`, value has `scenarios.length === 2`
  - `parseConfigJson('{')` → `ok: false`, error.message non-empty
  - `parseConfigJson('{"scenarios": [], "teams": []}')` → `ok: false` (structural check: scenarios empty)
  - `parseConfigJson('{"foo": 1}')` → `ok: false` (missing scenarios/teams)
  - `parseConfigJson('{"scenarios": [{}], "teams": [{}]}')` → `ok: true` (we don't deep-validate)
  - For a deliberately malformed string with a known error position, assert the line/col is correct

Props:
```ts
interface JsonEditorProps {
  value: string
  onChange: (next: string) => void
  errorLine?: number  // 1-based; if provided, highlight that line in the gutter
  ariaLabel?: string
}
```

Structure (per RESEARCH.md recommendation: plain textarea + flex-row CSS gutter, scroll-synced):
```jsx
<div class="relative flex h-full min-h-[400px] overflow-hidden rounded-lg border border-border-default bg-bg-panel font-mono text-sm">
  <pre ref={gutterRef} class="select-none bg-bg-surface px-3 py-3 text-right text-text-muted overflow-hidden">
    {/* one number per line of value */}
  </pre>
  <textarea
    ref={textareaRef}
    value={value}
    onChange={(e) => onChange(e.target.value)}
    onScroll={(e) => { gutterRef.current.scrollTop = e.currentTarget.scrollTop }}
    spellCheck={false}
    class="flex-1 resize-none bg-transparent px-3 py-3 text-text-primary outline-none"
    aria-label={ariaLabel ?? 'Configuration JSON'}
  />
</div>
```

Implementation notes:
- Compute line numbers via `value.split('\n').length`. Render as `1\n2\n3\n...` inside the `<pre>`.
- Use `useRef` for both elements; sync scroll on the textarea's `onScroll` handler.
- The `<pre>` and `<textarea>` MUST share identical `font-family`, `font-size`, `line-height`, and vertical padding so line numbers align with text rows. Use `font-mono leading-6 text-sm py-3` on both.
- If `errorLine` is provided, render that line number with a coloured background in the gutter (e.g. wrap each line number in a span and conditionally apply `bg-[var(--color-category-crisis)]/30 text-text-primary`).
- Tab key inside the textarea: leave default behaviour for now (tab moves focus). Phase 4 is not aiming at IDE parity; capturing tab-to-indent can wait.
- Component height: `h-full min-h-[400px]` — let the parent control overall sizing.

Do NOT install Monaco, CodeMirror, or any other editor library. Plain textarea is the locked decision.

**LoadConfigPanel** (`src/components/setup/LoadConfigPanel.tsx`):

State + hooks:
- Read `configJson`, `setConfigJson`, `setSetupMode`, `initGame` from `useGameStore`. Use the hook with selectors to avoid unnecessary re-renders.
- Local state: `parseResult: ReturnType<typeof parseConfigJson> | null`. Initialise by parsing the initial `configJson` once on mount (via `useEffect` with empty deps OR `useState(() => parseConfigJson(configJson))`).
- `useNavigate()` from `"react-router"`.

Debounced parse (per RESEARCH.md decision):
```tsx
useEffect(() => {
  const id = setTimeout(() => {
    setParseResult(parseConfigJson(configJson))
  }, 300)
  return () => clearTimeout(id)
}, [configJson])
```
The parse result lives in LOCAL component state, not Zustand. The store only holds the raw `configJson` string.

Layout (two-column at lg, single column at md and below):
```jsx
<main class="min-h-screen bg-bg-base text-text-primary p-6">
  <header class="mb-4 flex items-center justify-between">
    <button onClick={() => setSetupMode('home')} class="text-sm text-text-secondary hover:text-text-primary">← Back</button>
    <h1 class="text-xl font-semibold">Load Configuration</h1>
    <div /> {/* spacer */}
  </header>
  <div class="grid gap-6 lg:grid-cols-2">
    <section class="flex flex-col gap-3">
      <label class="text-sm text-text-secondary">Configuration JSON</label>
      <JsonEditor
        value={configJson}
        onChange={setConfigJson}
        errorLine={parseResult && !parseResult.ok ? parseResult.error.line : undefined}
        ariaLabel="Game configuration JSON"
      />
      {/* Validation error inline message — minimal in this plan; plan 04-04 polishes */}
      {parseResult && !parseResult.ok && (
        <p role="alert" class="text-sm text-[var(--color-category-crisis)]">
          Line {parseResult.error.line}, col {parseResult.error.col}: {parseResult.error.message}
        </p>
      )}
    </section>
    <section>
      {parseResult?.ok ? (
        <>
          <ScenarioSummary config={parseResult.value} />
          <div class="mt-4 flex flex-wrap gap-3">
            {parseResult.value.scenarios.map((s, i) => (
              <button
                key={s.id}
                onClick={() => handleLaunch(i)}
                class="rounded-lg border border-border-default bg-bg-panel px-4 py-2 hover:border-border-muted"
              >
                Launch Scenario {i + 1}
              </button>
            ))}
          </div>
        </>
      ) : (
        <p class="text-sm text-text-secondary">Fix the JSON to see the scenario summary.</p>
      )}
    </section>
  </div>
</main>
```

Launch handler:
```tsx
const handleLaunch = (scenarioIndex: number) => {
  // Re-parse at click time — debounced state may be stale (300ms window)
  const fresh = parseConfigJson(configJson)
  if (!fresh.ok) {
    // Sync local error display in case user clicked during the debounce window
    setParseResult(fresh)
    return
  }
  initGame(fresh.value, scenarioIndex)
  navigate('/game')
}
```

DO NOT clear `configJson` after launch — the user may want to come back. `resetGame` already handles cleanup.
