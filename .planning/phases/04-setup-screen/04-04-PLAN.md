  <files>src/components/setup/LoadConfigPanel.tsx</files>
  <action>
Update `src/components/setup/LoadConfigPanel.tsx` to elevate the error UX:

1. **Replace the disappearing Launch buttons with disabled buttons.** In 04-03 the buttons render only when `parseResult?.ok`. Change the logic so the buttons ALWAYS render when a parse result exists, but with `disabled={!parseResult?.ok}` and visual treatment:
   ```tsx
   <button
     key={s.id}
     onClick={() => handleLaunch(i)}
     disabled={!parseResult?.ok}
     aria-disabled={!parseResult?.ok}
     title={parseResult?.ok ? undefined : 'Fix JSON errors to launch'}
     class={cn(
       'rounded-lg border px-4 py-2 transition',
       parseResult?.ok
         ? 'border-border-default bg-bg-panel hover:border-border-muted'
         : 'border-border-subtle bg-bg-panel/50 text-text-muted cursor-not-allowed'
     )}
   >
     Launch Scenario {i + 1}
   </button>
   ```
   If you don't have a `cn` helper, inline the conditional class string. When parse hasn't yet produced a result (initial mount briefly), buttons may be hidden — that's fine, but once `parseResult` is set they are always rendered.

   To render scenario buttons even when parse fails, you need scenario count — fall back to deriving from the LAST KNOWN successful parse, or to a sensible default of 2 (matching EDIP). Implementation: keep a `useRef` (or local state) holding `lastValidScenarioCount: number | null` updated whenever `parseResult.ok === true`. When parseResult is invalid, render that many disabled buttons. If never valid, render nothing — that's an acceptable edge case (user pasted garbage from the start).

2. **Upgrade the inline error alert.** Replace the single-line `<p role="alert">` with a structured block:
   ```tsx
   <div role="alert" class="rounded-md border border-[var(--color-category-crisis)]/50 bg-[var(--color-category-crisis)]/10 p-3">
     <div class="text-sm font-semibold text-[var(--color-category-crisis)]">JSON parse error</div>
     <div class="mt-1 text-sm text-text-primary">{error.message}</div>
     <div class="mt-1 text-xs text-text-secondary">Line {error.line}, column {error.col}</div>
   </div>
   ```
   Place this directly under the JsonEditor (above the editor is also acceptable — pick whichever the JsonEditor's flex column places more visibly without scrolling).

3. **Keep `errorLine` prop wired to JsonEditor** so the gutter highlight remains. Do not change JsonEditor itself.

4. **Edge case — empty textarea:** If the user deletes everything, `JSON.parse('')` throws `SyntaxError: Unexpected end of JSON input`. This is fine — the existing parser handles it. But make sure the alert message reads sensibly (it will — the native error message is clear). No special-casing needed.

5. **Edge case — Launch click during the 300ms debounce window after a fix:** The Task 3 launch handler in 04-03 re-parses synchronously at click time, so a recently-fixed JSON works immediately even though the displayed `parseResult` may be stale (still showing error). With Launch buttons now ALWAYS visible, this can be a UX trap: button is visually disabled, user clicks anyway (some browsers allow click on disabled-styled-but-not-disabled-attr). Belt-and-braces: keep the `disabled` HTML attr in sync with `parseResult?.ok` — if it's truly stale, the user just waits 300ms. This is acceptable.

Do NOT add a confirmation modal. Do NOT add a "validate now" button. The 300ms debounce is the validation cadence.

```tsx
import { describe, it, expect, beforeEach, vi } from 'vitest'
import { render, screen } from '@testing-library/react'

import { MemoryRouter } from 'react-router'
import LoadConfigPanel from './LoadConfigPanel'

describe('LoadConfigPanel', () => {
  it('disables Launch buttons when JSON is invalid', async () => {
    useGameStore.setState({ configJson: '{ broken json' })
    render(<MemoryRouter><LoadConfigPanel /></MemoryRouter>)

    // Wait 350ms for the debounced parse to set parseResult to error
    await new Promise(r => setTimeout(r, 350))

    const buttons = await screen.findAllByRole('button', { name: /Launch Scenario/i })
    // May be 0 if no last-valid-scenario-count established — assert via the alert instead in that case
    if (buttons.length > 0) {
      buttons.forEach(b => expect(b).toBeDisabled())
    }
    expect(screen.getByRole('alert')).toBeInTheDocument()
  })
})
```

If `@testing-library/user-event` is not installed, use `fireEvent.change` directly. Don't add a new dep — `@testing-library/react` provides `fireEvent`.
