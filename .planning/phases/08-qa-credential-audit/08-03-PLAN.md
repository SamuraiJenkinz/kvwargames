    ## 4. Key Rotation Checklist

    **Background:** STATE.md line 218 records that the OpenAI API key appeared in chat logs across two smoke tests (Phase 6 and Phase 7). The key has since been scrubbed from `.env`, but the operational pattern that allowed it to leak was not codified. This checklist exists to prevent re-leak on the next rotation.

    ### When you rotate the API key:
    1. **Revoke the old key** in the LLM provider dashboard before generating the new one (not after — minimizes the window where the leaked key remains valid).
    2. **Update `backend/.env`** — replace `LLM_API_KEY=<old>` with `LLM_API_KEY=<new>`. Confirm `backend/.env` is still gitignored (`git check-ignore -v backend/.env` returns the .gitignore line).
    3. **Restart the dev backend** — `cd backend && uvicorn app.main:app --reload` (or however `pnpm dev:backend` is wired). The lifespan re-reads settings on startup; a hot-reload alone does NOT re-read env.
    4. **Verify the new key works** — single `POST /api/llm` test request from the frontend; confirm 200 response.
    5. **Scrub any pre-rotation transcripts** — search `.planning/phases/**/*.md` for the old key prefix (`grep -r "<old-key-prefix>" .planning/`); redact if found.
    6. **Run this audit doc's grep commands again** — confirm zero matches in `src/`. The new key MUST NOT have crept into source.
    7. **Log the rotation** in STATE.md ops history (date + reason).

    ### Why this exists in this audit
    The leak vector in Phases 6 and 7 was not a code defect — it was an operational gap. Operators pasted DevTools console output (which includes outbound request bodies in some configurations) into smoke-test summaries. This checklist makes the rotation cycle explicit; future operators can follow it without rediscovering the trap.
