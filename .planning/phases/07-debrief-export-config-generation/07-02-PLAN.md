<objective>
Wire the debrief export button into the live game UI, split `triggerDebrief` semantics into interim vs end-of-game, and gate the send/advance buttons on `gameEnded`. End-to-end path: facilitator clicks `End Game + Debrief` → divider inserted → LLM debrief turn → persona responses → Download button appears → click produces `.md` file.

Purpose: Satisfies DEB-01, DEB-02, DEB-03 end-to-end. Plan 07-01 built the exporter primitives; this plan makes them reachable from the UI.

Output: ActionToolbar gains a Download Debrief button (conditional on debrief_divider presence), `End Game + Debrief` uses a new `endGame()` store action that sets `gameEnded=true`, and the FacilitatorInput Send button / round-advance buttons honour the `gameEnded` gate.
</objective>

<decisions_locked>
1. **Two distinct store actions**: keep `triggerDebrief()` as the interim debrief (existing behaviour) and add a NEW `endGame()` action that sets `gameEnded=true` and INLINE-fires the debrief LLM turn (divider push + loading=true + runLLMTurn call, all duplicated from triggerDebrief's body). Rationale: minimises risk to existing tests (22+ tests depend on `triggerDebrief` semantics); `endGame` is an additive, single-purpose verb.
2. **Download button visibility**: always rendered in ActionToolbar layout but hidden until `messages.some(m => m.type === 'debrief_divider')` is true. Use conditional render (`{showDownload && <button>...</button>}`) — avoids flashing disabled→enabled state.
3. **Download is idempotent**: each click regenerates from current store snapshot. No caching, no stored history.
4. **`gameEnded` gates FOUR UI primitives** (per CONTEXT.md decisions_locked item 4): Send button (`disabled={loading || gameEnded}`), "Advance to Round N" button (disabled or hidden), "Request Debrief Now" button (disabled — post-game interim debrief is nonsensical), AND "End Game + Debrief" button itself (disabled — cannot end twice).
5. **No confirmation dialog** for either download or End Game — CONTEXT.md explicitly rules out confirmations. End Game is not destructive (no data is lost; it just flips a flag).
6. **Download snapshot read**: construct `DebriefSnapshot` fresh from `useGameStore.getState()` inside the onClick handler (not via selector hook) — avoids re-render storms when Download button is visible. Use `getState()` once per click.
</decisions_locked>

    ```ts
    endGame: () => {
      // Bail if already loading or already ended (idempotent — second click is a no-op).
      if (get().loading || get().gameEnded) return

      const prefixedInput = '[END_GAME_DEBRIEF] Facilitator ended the game.'

      // Delegating to triggerDebrief() would bail on its gameEnded guard once we
      // flip the flag — inline the divider push + LLM kick-off here. Keep in sync
      // with triggerDebrief()'s divider-push block if that ever changes.
      set((s) => {
        s.gameEnded = true
        s.messages.push({
          id: crypto.randomUUID(),
          type: 'debrief_divider',
          label: 'DEBRIEF',
          timestamp: formatTime(),
          isDebrief: true,
        })
        s.loading = true
        s.lastFacilitatorInput = prefixedInput
      })
      void runLLMTurn(prefixedInput)
    },
    ```
