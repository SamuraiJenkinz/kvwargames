## E2E Flows — Traced

### Flow A: Launch-time health gate ✅

1. Facilitator navigates to `/setup` → SetupScreen routes to LoadConfigPanel
2. Panel mounts with `healthStatus='checking'`; HealthBadge `useEffect` fires `fetch('/api/health/llm')`
3. Backend probes upstream LLM with 15s SLA (per-request timeout override at `health.py:103`); returns `{ok, latencyMs}` or `{ok:false, code, hint}`
4. On `ok:true` → badge shows green dot + "Connected — 820ms"; `healthStatus='ok'`; Launch enabled
5. On `ok:false` → red dot + code/hint; `healthStatus='failed'`; Launch disabled with tooltip "LLM health check must pass before launching"
6. Re-check button re-runs probe; aborts any in-flight request first

**Break points checked:** none found. Gate is tight.

### Flow B: Null-state recovery ✅

1. User hits `/game` with `gameState === null` (e.g. bookmark, hard refresh without setup)
2. `GuardedGameScreen` reads store, sees null, returns `<Navigate to="/setup" replace />`
3. Browser URL becomes `/setup`; no history entry for `/game`
4. No blank screen, no console warning (SetupScreen mounts cleanly)

**Break points checked:** none. `replace` flag prevents back-button bounce loop.

### Flow C: Crisis escalation in live loop ✅

1. Finch response with `stateUpdate: { crisisSeverity: 4, crisisState: "Security-Related Supply Crisis" }` returned by LLM
2. `parsePersonaResponse` validates shape — passes (speaker, message, stateUpdate object, flag null/string)
3. `applyStateUpdatePure` clamps `crisisSeverity` to 0–5 (still 4) and sets `crisisState` unclamped (enum pass-through per stateUpdater comment L93)
4. Store wholesale-replaces `gameState` → Zustand notifies subscribers
5. `GameHeader` re-renders `CrisisBadge` with red classes; chat feed shows Finch persona message via `buildPersonaMessage`

**Break points checked:** none. All four transformation stages preserve `crisisState`.
