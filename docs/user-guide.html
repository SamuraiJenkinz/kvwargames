KV WarGame User Guide

Facilitator handbook · Version 1.2

A quiet instrument for a loud table.

This guide walks you through everything the tool does, in the order you are likely to use it. Read it once end-to-end before your first live session, and keep it open in a second tab during the session as a reference. If you only have five minutes, use the quick start instead.

What this tool is

Chapter 01 · Mental model

KV WarGame is a facilitation console for live policy tabletop exercises. It runs alongside you — not in place of you — during a session of the EDIP Security of Supply Wargame or any scenario you load into it.

The central idea is simple: three AI personas listen to what you describe at the table and respond in character. The tool also tracks game state — crisis severity, legitimacy, and each team’s resources — and keeps the fiction coherent as the scenario escalates. You run the room. The tool listens, speaks, and remembers.

What it is not

  • It is not a player-facing tool. Players are at a physical table.
  • It is not a game engine that plays the scenario for you. You advance rounds, you play cards at the table, you make the calls.
  • It is not multi-user. One facilitator operates the app on a laptop for each session.
  • It does not persist anything between sessions. Close the tab and the game is gone. Export the debrief before you leave.

The three personas

Chapter 02 · Who speaks, and when

Each persona has a distinct voice, domain, and a set of triggers the tool uses to decide who should speak. You will come to recognise them quickly by their colour, their tone, and the kinds of facts they reach for.

Chair · Framing & Legitimacy
Kent Valentina

“Let us pause on the process here. Before we decide, we should be clear what we are deciding and on what authority.”

Speaks when Round starts · disputes · institutional or political card plays · debrief framing.
Analyst · Evidence & Consequence
Alistair Finch

“On the numbers — Team B’s readiness falls below threshold in two rounds if this continues. That is consequential.”

Speaks when Round inject delivery · adversary or disruption cards · escalation events · threshold warnings.
Engineer · Process & Operations
Michael Chen

“Operationally, the supply line runs through three nodes — if any one of them drops out, the whole flow collapses.”

Speaks when Technical or operational card plays · readiness changes · procedural challenges · debrief lessons.
On ordering When multiple personas speak in the same turn, they always appear in the order Kent → Finch → Chen. This is enforced by the tool, not by the model — the fiction stays stable even if the AI returns them out of order.

The setup screen

Chapter 03 · Getting to a launched scenario

The first screen you see offers two paths.

Load Config · EDIP Default

The standard path. Opens a JSON editor pre-populated with the EDIP Security of Supply Wargame — two scenarios, four teams, eleven EDIP cards, four national actions. On the right you see a live summary of the parsed config. Below the editor are Launch Scenario 1 and Launch Scenario 2 buttons. Clicking one initialises the game and takes you to the game screen.

You can edit the JSON directly — change team names, adjust starting resource values, rewrite the inject text — and the summary will update after a brief debounce. If you break the JSON, a red error note below the editor tells you the line and column of the problem, and the Launch buttons grey out until the JSON parses again.

Generate from Brief

For when you want a custom scenario. A free-text field accepts a description of the situation you want to simulate — a domain, a set of actors, a triggering event. The tool calls the LLM, produces a full JSON config matching the EDIP shape, and drops it into the same editor as above so you can review and edit before launching.

Generated configs are validated field-by-field. If the model produces a team with a PC value outside the allowed range, the editor shows an amber banner naming the exact field — e.g. teams[2].pc: expected number 0-6, got 9 — and disables the Launch buttons until you correct it (or re-generate).

First time For your first session, always use the EDIP default. The canonical scenario has been playtested. Custom briefs are a v1 feature but reward familiarity with the base game.

The game screen

Chapter 04 · Anatomy of the interface

The game screen is three columns with a sticky header and a sticky input bar. Every element has a purpose; nothing is decorative.

Left State panel — severity, legitimacy, teams
Centre Chat feed — facilitator and persona messages
Right Reference — cards, actions, guide

The header

Sticks to the top of the screen. Shows the app wordmark, the game name, the scenario, the round counter, and the current crisis state as a coloured badge. A New Game button on the right returns to setup and resets the store.

The facilitator input

Sticks to the bottom. On the left are action buttons — Advance to Round N, Request Debrief Now, and when applicable End Game & Debrief. On the right is a text field for typing events. Enter sends; Shift + Enter adds a new line. While the AI is thinking, the input and all action buttons are disabled.

Message types

Chapter 05 · Reading the chat feed

The feed renders seven kinds of message. Recognising them quickly is the single most useful facilitation skill.

Persona
A named AI voice, in character. Coloured avatar with the persona’s initials (KV / AF / MC), the persona name, a timestamp, and a 2 – 4 sentence message in a tinted bubble. An amber inline flag underneath marks something for your attention — e.g. “Note: Team B at PC threshold”.
Facilitator
What you typed. Right-aligned dark bubble with the label FACILITATOR and a timestamp. Shows exactly what the AI received — useful when a response surprises you.
Round divider
A round transition. Full-width horizontal line with a round label badge in the centre. Written into the feed when you press Advance to Round N.
Debrief divider
The closing bell. An amber-coloured variant of the round divider that marks the start of the debrief phase. The Download Debrief button becomes available after this appears.
Error
Something went wrong with the AI call. Red-tinted bubble with an error code, a short explanation, a collapsible disclosure showing the raw AI response, and a Retry button. See Chapter 09 for what the common codes mean.
Loading
The AI is thinking. A short animated three-dot indicator. Disappears when the response arrives or an error is raised.
Control banner
The AI has flagged a game-control decision for you. Appears below the facilitator input as an amber banner when the AI signals e.g. a threshold crossing that deserves explicit facilitator acknowledgement. Confirm or dismiss to continue.

Running a session

Chapter 06 · The moves you will make

Describing events

Treat the input bar as a stage direction, not a prompt. Short, concrete, in-world descriptions work best. Good: “Team C plays Emergency Stockpile Release. Team A objects on legitimacy grounds.” Less good: “What happens if a team does a stockpile thing?”

The personas will decide who responds based on the content. Card plays usually produce one or two voices; disputes and debriefs tend to produce all three.

Advancing a round

When the table is ready to move forward, press the Advance to Round N button. The feed shows a round divider, then Finch delivers the round’s inject, and Kent frames the key tension for discussion. The round counter in the header increments.

Requesting a debrief mid-game

The Request Debrief Now button gives you an interim pulse-check — all three personas reflect on the session to this point without ending it. Useful for longer games or when the table seems stuck.

Cards and national actions

You do not click cards in the interface — you play them at the physical table and describe the play in the input. The tool routes to the right persona based on the card’s category. Use the reference panel to confirm the correct card name, category, and effect before announcing a play.

Rhythm A healthy session has the facilitator speaking, the AI replying, and the table responding — in that order. If the AI feels like it is leading, slow down and describe more of what has just happened at the table before advancing.

Live game state

Chapter 07 · Reading the left-hand panel

The state panel is the live dashboard. All values update automatically from AI responses — you do not edit them directly.

Crisis severity
A track bar from 0 to 5, shown in red. Represents how acute the crisis is right now. Transitions up or down as the fiction develops.
EDIP legitimacy
A centre-zero track bar from −2 to  +2, shown in blue. Represents institutional credibility of the EDIP response. Positive is better; negative signals erosion.
Team resource grid
One card per team, each with six fields: PC (political capital), PO (political outlook), RDY (readiness), STK (stock), CRM (crisis reserve materials), IC (industrial capacity). Values are clamped to valid ranges silently — the AI cannot push a value out of bounds.
PC warning badge
Appears on a team card when political capital is low. STRAINED (amber) at PC = 1; CRISIS (red, blinking) at PC = 0. A crisis team cannot act until it recovers.
Persona indicators
Three coloured dots with names at the bottom of the panel. Indicate which personas are “active” for the current round — a subtle signal that changes across a scenario.
Silent clamping If the AI attempts to set a value outside its valid range — for example, pushing crisis severity to 7 — the tool clamps to the maximum silently. No error is shown to you. This is by design; the fiction continues, and the state stays coherent.

The reference panel

Chapter 08 · Right-hand tabs

Three tabs. Scroll position is preserved per tab, so you can flip between them without losing your place.

Cards

A list of all EDIP cards in the loaded scenario, grouped visually by category colour (crisis state, monitoring, prioritisation soft / hard, demand coordination, production acceleration, transfers). Click a card to see its full detail: ID, category, name, timing, requirements, and effect.

Actions

Two sections. National Actions — the four actions any team can take, with name, summary, and political cost. Unique Team Powers — one ability per team, indexed by the team letter (A, B, C, D).

Guide

The facilitator’s own reference: the objective, red lines and PC thresholds, the voting rule (Support ≥ 3, Objections ≤ 1), resource tokens, the EO response mechanic, and input guidance. Skim it at the start of each session.

When something goes wrong

Chapter 09 · Error states and recovery

The tool is designed to fail without losing your session. In practice this means you will occasionally see red bubbles and should know what they tell you.

Red error bubble

The AI returned a response the tool could not parse, or the backend could not reach the LLM. The bubble shows an error code in brackets — for example [PARSE_FAILURE], [LLM_TIMEOUT], [LLM_UNREACHABLE] — followed by a short explanation. A disclosure toggle reveals the raw response (for technical review). A Retry button replays your last facilitator message with a fresh AI call.

What to do

  • Click Retry. This works most of the time.
  • If retry fails twice, rephrase your last message and send it again — sometimes the model gets stuck on an ambiguous prompt.
  • If all retries fail with LLM_UNREACHABLE, the backend or the LLM endpoint is down. Tell your IT contact; the session state is preserved, and you can resume when the service returns.
Session preservation A failed turn never mutates game state or history. Team resources, messages prior to the error, and the round counter are all intact.

Control banner

Occasionally the AI surfaces a structured request for your decision rather than responding in persona — for example, flagging that a resource change crosses a critical threshold. An amber banner appears above the input bar with Confirm and Dismiss. Neither is destructive; choose based on the table’s intent.

Ending and exporting

Chapter 10 · Close the session properly

When the scenario is complete — either because you have run all rounds or the table has reached a conclusion — press End Game & Debrief. A debrief divider appears, all three personas give structured closing remarks, and a Download Debrief button becomes available in the action toolbar.

Clicking Download Debrief saves a Markdown file named debrief-<game-name>-<date>.md to your downloads folder. It contains:

  • Game metadata (name, domain, scenario, date, rounds played).
  • Per-round sections with the state snapshot at round start and the full transcript, persona-attributed.
  • The debrief transcript.
  • A final-state team table.
  • An appendix with the raw JSON config used for the session.
Ephemeral by design Closing the browser tab ends the session completely. There is no history, no autosave, no cloud backup — this is intentional for the sensitivity of the content. Download the debrief before you leave.

Debrief podcast (optional)

After the debrief lands, a Generate Podcast button appears in the action toolbar next to Download Debrief. Clicking it has each persona read their own closing remarks through a distinct voice — Kent, then Finch, then Chen — stitched into a single MP3 with ~700 ms of silence between segments. No new AI conversation is started; the personas read the debrief text that is already on the screen.

A player appears inline once generation completes. It carries three Skip to Kent / Finch / Chen buttons, a Now playing label that updates at segment boundaries, a Download MP3 button (filename debrief-<game-name>-<date>-<time>.mp3), and an expandable transcript panel below the player for reference. The audio loads paused — nothing auto-plays into the room.

Long debriefs (roughly more than two thousand words across all three personas) trigger a confirmation dialog that shows the estimated audio length before the request is sent upstream. Clicking Generate a second time with the same text returns the previously-rendered MP3 from cache with no additional network call. A dedicated Re-generate button invalidates the cache after confirming.

If the podcast is unavailable A small TTS badge on the setup screen reports whether the podcast service is reachable — it is informational only and never prevents Launch. If the badge is amber, or if generation fails mid-stream, a red banner names the reason and the adjacent Download Debrief button continues to work normally. The two paths do not share a failure boundary by design.

Security posture

Chapter 11 · A summary for your IT contact

The tool runs on a server inside your corporate network. Your browser calls only that server; it never talks to an external LLM vendor directly. All LLM credentials — API keys, endpoint URLs, extra headers — live in a server-side environment file that is readable only by the service account running the app.

Every in-session AI call is proxied through the internal server, which injects credentials in the outbound request. Your browser network tab contains no Authorization header, no API key, no upstream URL. A Phase 8 credential audit (HAR capture) confirms this.

The debrief-podcast feature (v1.2) extends the same posture to ElevenLabs. When real voices are enabled, the ELEVENLABS_API_KEY lives alongside LLM_API_KEY in the server-side environment file and is injected into outbound requests the same way; the browser never sees it. The podcast audio travels from the server to the browser as a blob; no ElevenLabs URL is ever fetched directly by the client. When real voices are not configured, the tool ships with a "fake" TTS provider that uses local audio fixtures and makes zero outbound calls.

The tool stores nothing between sessions. There is no database, no cookie, no localStorage for game content. The exported debrief Markdown file (and, if you choose to save it, the stitched MP3) is the only persistent artifact, and it lives wherever you save it.
